## Context

Hypso is a native Rust desktop app (Slint GUI, resvg raster) that generates
seed-reproducible topographic wallpapers. Its generative core is layered so the
pure geometry (`noise` → `contour` → `smooth`, over `geom`) has **no dependency
on resvg/slint/fastrand** — it uses only `std` and a seeded PRNG. The sibling
project `terminal-webcam` already establishes a proven GitHub Pages pattern:
sources in `web/`, minified into `dist/`, deployed by an Actions workflow, fully
self-contained (no CDN), with strong SEO/a11y and a browser demo of the tool
itself.

This design reuses both facts: the pure core compiles to WASM cleanly, and the
`terminal-webcam` layout is the template. The core invariant of Hypso —
`svg::emit(&Config) -> String` as single source of truth for the desktop
pipeline — must not be disturbed.

## Goals / Non-Goals

**Goals:**
- Publish a conceptual landing page that *is* the product: the real engine
  running in the browser via WASM, animated with GSAP.
- Keep the page fully self-contained and offline-capable (GSAP vendored).
- AAA contrast, complete SEO, and robust graceful degradation
  (reduced-motion / no-WASM / no-GSAP → static poster).
- Generate the OG image and posters with Hypso's own pipeline (`examples/site.rs`).
- Additive-only Rust changes; native build, tests, and coverage gates untouched.

**Non-Goals:**
- No feature list, docs site, or interactive full editor in the browser.
- Not compiling the full resvg/Slint pipeline to WASM (heavy, unnecessary).
- No runtime dependency on npm/bundlers or third-party CDNs.
- No changes to `svg::emit` or the desktop render path.

## Decisions

### D1 — Live hero runs the real Rust core compiled to WASM
The hero calls the actual `noise` + `contour` + `smooth` functions compiled to
`wasm32-unknown-unknown`. This is the strongest expression of the project and
guarantees browser output matches the app.
- *Alternative — reimplement marching-squares/fBm in JS*: lighter, no toolchain,
  but it is a second implementation that can drift and is not "Hypso running".
  Rejected.
- *Alternative — GSAP morphing between Rust-pregenerated SVG frames*: no live
  generativity, fixed loop. Rejected as the primary path (kept only as the
  poster/fallback source).

### D2 — Binding surface: thin `wasm-bindgen` module, target-gated
Add a small binding (e.g. `src/wasm.rs` compiled only for the wasm target, or a
dedicated `--target wasm32` lib target) exposing one entry point:
`generate(seed, field_size, levels, frequency) -> flat polyline buffer`.
Return normalized `[0,1]` coordinates as a flat `Float32Array` + offsets, so JS
draws without per-point marshaling cost.
- `wasm-bindgen` is added as a **target-gated / optional** dependency so the
  desktop binary never links it.
- *Alternative — expose `render_png` over WASM*: would drag resvg+fonts into
  WASM (megabytes, slow). Rejected; the browser draws the polylines itself on a
  `<canvas>`/SVG, which is what makes the stroke-draw animation possible.

### D3 — Drawing + animation: Canvas 2D (or SVG paths) + GSAP
JS receives polylines and draws them; GSAP drives the stroke-draw (via
`strokeDashoffset` on SVG paths, or incremental canvas path reveal), seed drift
(crossfade/morph between two generated fields), and pointer parallax (layered
transforms). GSAP is **vendored** in `web/js/` (self-host permitted by its free
license) → no CDN, CSP-safe.
- *Alternative — CSS/WAAPI only*: insufficient control for orchestrated
  stroke-draw + timeline sequencing the brief demands ("use e abuse de GSAP").

### D4 — Static assets generated by Hypso (`examples/site.rs`)
Mirror `gallery.rs`: use `render::render_png` to emit the **1200×630 OG image**
(dark cartographic composition with the title in an AAA-contrast reserved text
zone) and one or more **poster** PNGs used as the reduced-motion / fallback hero.
Reproducible, committed, single `cargo run --example site`.

### D5 — Build & deploy mirror `terminal-webcam`, extended for Rust/WASM
Keep `web/ → docs/` minification. The CI site job additionally: installs the
wasm target + `wasm-bindgen`/`wasm-pack`, builds the WASM module into the web
asset tree, runs `cargo run --example site` to (re)generate OG/posters, minifies
into `dist/`, and deploys via `actions/deploy-pages`. Triggers scoped to
`web/**`, the WASM source, `examples/site.rs`, and the workflow file.
- The `dist/` output may be committed (like the reference) or built fresh in CI;
  building fresh in CI avoids committing large binaries. Chosen: build in CI,
  commit only `web/` sources. (Confirm during apply.)

### D6 — Aesthetic: dark cartographic / instrument
Near-black background, contour lines in cyan/amber (reuse the `midnight`/`ocean`
sample palette and the logo's `#6fb6c8`), technical mono type, a subtle HUD
(seed / coordinates) as texture. Poster and OG share this palette for a coherent
share-preview. AAA contrast verified for all text.

## Risks / Trade-offs

- **WASM toolchain in CI adds complexity** → Pin `wasm-bindgen` version to the
  crate's `wasm-bindgen` dep; use `wasm-pack` or `cargo` + `wasm-bindgen-cli`
  with a locked version to avoid version-skew panics.
- **WASM payload / first-paint latency** → Keep the binding tiny (pure geometry
  only, no resvg); `strip`/`opt-level=z` for the wasm profile; show the static
  poster immediately and swap to live once the module resolves (no layout shift).
- **GSAP licensing when self-hosting** → Use the free/standard GSAP core (MIT-
  equivalent no-charge license) and vendor only what's used; document the source
  version. Avoid Club/premium plugins.
- **Determinism drift across the boundary** → Add a parity check: same
  seed/params through native `contour` vs WASM must match; guard in the WASM
  crate's own tests.
- **Accessibility of a generative canvas** → Provide `aria-label` + offscreen
  text description; ensure the CTA and content are fully usable without the hero.
- **Repo/name mismatch** → Cargo `repository` is `github.com/skvggor/hypso`;
  confirm the Pages URL (`https://skvggor.github.io/hypso/`) and base path before
  wiring canonical/sitemap/manifest `start_url`.

## Migration Plan

Purely additive; nothing to migrate. Rollback = revert the `web/`, `dist/`,
`examples/site.rs`, WASM module, and workflow additions. The desktop app and its
release pipeline are unaffected. Deployment is idempotent via the Pages workflow;
reverting the workflow stops future deploys.
